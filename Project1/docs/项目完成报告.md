# 项目完成报告（第一阶段）

## 1. 项目目标

**本项目目标**是实现一个基于 MongoDB 的在线问卷系统第一阶段版本，保证"**可建模、可运行、可验证、可追溯**"具体目标如下：

1. **完成用户注册登录、问卷创建、题目配置、跳转配置、填写提交、统计分析的功能**
2. **使用 MongoDB 建立符合问卷业务特征的数据模型**，支持灵活扩展题型与规则
3. **在后端实现严格业务约束**，避免"前端可绕过"的数据脏写风险
4. **用完整文档和测试证明系统可复现、可验收**
5. **规范记录 AI 参与过程**

## 2. 系统设计

### 2.1 技术架构

1. **前端**：Vue3 + Vite + Pinia + Vue Router + Axios + TailwindCSS
2. **后端**：Django + DRF + PyMongo
3. **数据库**：MongoDB
4. **鉴权**：JWT（Bearer Token）

### 2.2 前后端职责划分

1. **前端负责**：页面交互、表单收集、状态展示、接口调用
2. **后端负责**：鉴权、权限控制、规则匹配、业务校验、统计聚合
3. **MongoDB 负责**：业务数据持久化与索引加速

### 2.3 核心模块

1. **认证模块**：注册、登录、token 签发与校验
2. **问卷模块**：创建、发布、关闭、删除、列表查询
3. **题目模块**：单选/多选/填空三类题目维护，结果的校验
4. **跳转模块**：规则配置、规则匹配、下一题计算
5. **提交模块**：路径校验、必答校验、重复提交控制
6. **统计模块**：整卷统计、单题统计、提交记录摘要

### 2.4 安全与约束策略

1. **非公开接口**默认要求 JWT 登录
2. **编辑类操作**仅允许草稿问卷
3. **发布/关闭状态**不可回草稿
4. **重复提交是否允许**由问卷配置字段控制
5. **匿名提交**不暴露 respondent_username

## 3. MongoDB 设计

### 3.1 集合划分

1. **users**：用户账号信息
2. **surveys**：问卷主体信息
3. **questions**：题目与校验规则
4. **jump_rules**：跳转条件规则
5. **answers**：提交结果与明细答案

### 3.2 关键字段设计

1. **surveys.status**：draft/published/closed，驱动生命周期
2. **questions.validation**：统一承载题型校验参数
3. **jump_rules.priority**：多规则冲突时决定命中顺序
4. **answers.is_anonymous**：控制统计展示隐私
5. **answers.answers[]**：内嵌每题作答，便于一次读取整份提交

### 3.3 索引设计与命中场景

1. **users.username 唯一索引**：注册防重、登录快速定位
2. **surveys.slug 唯一索引**：公开问卷链接访问
3. **surveys.owner_id + created_at**：我的问卷列表查询
4. **questions.survey_id + order**：按顺序读取题目
5. **jump_rules.survey_id + question_id + priority**：跳转判定
6. **answers.survey_id + submitted_at**：统计读取
7. **answers.survey_id + respondent_id**：重复提交判重

## 4. 跳转设计

### 4.1 规则模型

每条跳转规则包含：

1. **question_id**：来源题目
2. **rule_type**：single_choice/multi_choice/number
3. **operator**：eq/in/contains_any/contains_all/gt/gte/lt/lte/range
4. **value**：比较值
5. **target_question_id 或 target_order**：跳转目标
6. **priority**：优先级（数字越小越优先）

### 4.2 执行流程

1. 前端调用 next-question，提交 current_question_id 与 answer
2. 后端读取该题所有规则并按 **priority 升序**处理
3. 依次进行条件匹配，**首条命中即返回**对应下一题
4. 若无命中，按题目 order 进入顺序下一题
5. 若已到末题，返回 next_question=null

### 4.3 设计收益

1. **分支路径可配置**，不需要改代码硬编码逻辑
2. **同一题支持多条件并发配置**，行为由 priority 控制
3. **可扩展性强**

## 5. 校验设计

### 5.1 题型校验

1. **单选题**：答案必须是 options 中的合法 key
2. **多选题**：答案必须是数组，且选择数量满足 min_select/max_select
3. **填空数字**：满足 min_value/max_value，is_integer=true 时必须为整数
4. **必答题**：required=true 的题目必须有答案

### 5.2 提交校验链路

提交接口采用"**先校验后入库**"，核心步骤如下：

1. **校验问卷状态**是否允许提交（已发布、未过截止时间）
2. **基于用户答案重算真实作答路径**
3. **检查提交答案**是否都来自可达路径
4. **执行题型值校验**与必答校验
5. **根据 allow_multiple_submissions**控制重复提交
6. **全部通过后写入** answers

## 6. AI 使用说明

### 6.1 AI 帮了什么

1. AI 对我设计的**数据库结构及索引**做了优化
2. AI 使用我设计的**后端逻辑**完成了前后端大多数功能的落地
3. AI 为我完成了**文档架构以及模板**的搭建

### 6.2 AI 做错了什么

1. AI 在前端设计中常有**布局问题**，甚至是**设计错误**，如：添加了根本用不上的按钮、在逻辑上不能点击按钮时仍然可以完成点击操作
2. AI 生成的后端逻辑中**时间比对有问题**，系统时间与问卷存储的截止时间格式不一致
3. **问卷编辑部分过于复杂**，不利于用户使用
4. AI **没有为用户注册添加用户名唯一校验与长度约束**
5. **匿名逻辑有问题**，允许使用匿名的问卷中没有为用户提供是否选择匿名的选项
6. 在问卷填写完成后，**用户可以反复提交**，还可以提交后返回问卷填写页面继续填写
7. **问卷的状态转换逻辑有问题**，在发布后可以进入草稿状态
8. 在一些禁止逻辑上**只让前端无法进行**，但是可以绕过前端通过 API 直接操作
9. AI 生成的**测试样例没有完全覆盖需求功能**，以及有部分测试重复了
10. AI 生成的**文档结构顺序有些混乱**，并且在生成的内容中存在**大量幻觉**

### 6.3 自己改了什么

1. AI 给出优化后数据库方案后，我**补充 allow_multiple_submissions 等字段**
2. **添加用户名唯一校验与长度约束**
3. **添加一些有关状态的限制**：问卷在发布和关闭后无法再被编辑，无法再次转换为草稿态，只有在问卷发布状态才能复制填写链接
4. **大量前端上的调整**：冗余按钮删除，冗余提示删除，布局优化等
5. **为问卷删除添加了一个二次确认**
6. **为跳转逻辑添加了优先级设计**
7. **修复了问卷回答完页面中出现的可以重复提交的问题**以及提交后还可以返回修改问卷回答的问题
8. **尝试修改了问卷统计功能中以 json 形式展示单题部分的问题**，复用了整卷展示部分的代码来完成这一修复（但没改好）
9. **修改、优化了文档内容**，将存在幻觉的部分删掉
10. **调整了测试样例**
11. 其实大多数自己改的内容交给 ai 也能很好地解决。。。但主要是想要熟悉一下开发，才自己来做的

### 6.4 自己设计了什么

1. **数据库结构以及索引**
2. **后端的大部分逻辑设计**，如跳转、校验等
3. **前端的一些布局设计**，如问卷填写情况总览，问卷填写结束页面等

## 8. 遇到的问题

1. **发布后编辑限制与"回草稿"等状态转换限制初期不完整**
2. 在数据库设计中**在一些表中是否添加冗余字段纠结**
3. **统计页最初无法区分实名与匿名提交用户信息展示**
4. **系统初期存在数据与校验规则需求的格式不一致导致错误**
5. **测试文档早期 API 测试部分输入不够详细**，难以直接复现
6. **前端状态操作按钮与后端约束不一致**（填写链接常亮引导误解）
7. **部分前端布局有问题**，存在盒子挤压、字体大小等问题
8. **问题面板大量报错**，但程序能够正常运行

## 9. 如何解决

1. **补齐后端状态机约束**：非草稿不可编辑，发布/关闭不可回草稿
2. **几乎没有使用冗余字段**，使用索引加速查询
3. **扩展统计接口 submissions 字段**：实名返回 respondent_username，匿名返回 null
4. **调整数据格式与函数逻辑保持一致**，消除伪失败
5. **重写测试文档 API 测试部分**：按步骤补全路径参数、Header、Body
6. **修改前端逻辑**，严格限制前后端逻辑一致，避免对用户的引导误解
7. **优化前端布局**，修复布局问题
8. **问题面板中的报错是 python 解释器的类型检查的报错**，实际上代码中有类型检查逻辑，不会导致应用运行出错

**最终效果**：

1. **功能实现、业务约束、测试结果、文档内容完全实现**
2. **项目达成作业第一阶段要求**，且具备后续阶段迭代基础

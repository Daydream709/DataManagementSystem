# 项目完成报告（第二阶段）

## 1. 项目目标

**第二阶段目标**是在第一阶段在线问卷系统的基础上，新增"**常用题库**"功能模块，支持题目的保存、复用、版本管理、共享与跨问卷统计，具体目标如下：

1. **用户可以保存常用题目**，方便以后重复使用
2. **可以把题目共享给其他用户**，其他用户可以在自己的问卷里使用这些题目，也可以收藏到自己题库或移除
3. **修改题库题目时不影响已经做好的问卷**，已发布问卷保持原有内容
4. **记录题目的修改历史**，可以看到版本关系，可以恢复旧版本
5. **同一个题目的不同版本可以同时存在**，不同问卷可使用不同版本
6. **可以查看某个题被哪些问卷使用**，包括收藏来源的问卷，修改时心中有数
7. **建立题目库**，做问卷时直接从题库选题，题库中的题可更新版本，编辑时支持修改题目内容（题型、选项、校验规则等）
8. **查看单个题目的跨问卷统计结果**，按指定版本统计同一题在所有问卷中的回答情况

## 2. 第一阶段设计回顾

### 2.1 第一阶段系统架构

1. **前端**：Vue3 + Vite + Pinia + Vue Router + Axios + TailwindCSS
2. **后端**：Django + DRF + PyMongo
3. **数据库**：MongoDB，共 5 个集合（users、surveys、questions、jump_rules、answers）
4. **鉴权**：JWT（Bearer Token）

### 2.2 第一阶段数据模型

第一阶段中，题目（questions）完全依附于问卷（surveys），每道题只属于一个问卷，题目之间没有关联，没有版本概念，没有共享机制。核心集合关系为：

1. users 1:N surveys
2. surveys 1:N questions
3. surveys 1:N jump_rules
4. surveys 1:N answers

### 2.3 第一阶段局限性

1. **题目无法复用**：每次做新问卷都要重新写一遍题目
2. **没有共享机制**：题目不能共享给其他用户
3. **没有版本管理**：题目修改后无法追溯历史
4. **没有跨问卷统计**：统计只能看单个问卷，无法看同一题在多个问卷中的汇总

## 3. 新需求带来的问题与设计调整

### 3.1 新需求带来的核心问题

| 问题编号 | 问题描述                                         | 涉及需求                     |
| -------- | ------------------------------------------------ | ---------------------------- |
| P1       | 题目需要一个独立的存储位置，脱离问卷存在         | 需求一、七（题库保存与管理） |
| P2       | 一道题被多个问卷使用时，修改不应影响已导入的问卷 | 需求三（修改不影响旧问卷）   |
| P3       | 题目的多次修改需要保留完整历史                   | 需求四（版本历史与恢复）     |
| P4       | 同一题的不同版本需要同时存在                     | 需求五（多版本共存）         |
| P5       | 题目需要能够共享给指定用户                       | 需求二（共享给同事）         |
| P6       | 需要追踪一道题被哪些问卷使用                     | 需求六（使用情况查询）       |
| P7       | 需要聚合同一题在多个问卷中的回答数据             | 需求八（跨问卷统计）         |

### 3.2 为什么需要修改原设计

第一阶段中，questions 集合中的题目是"问卷私有"的——每道题只属于一个问卷，没有跨问卷关联。新需求要求题目能脱离问卷独立存在、支持版本管理和跨用户共享，这意味着：

1. **需要新增一个独立的集合**来存储"脱离问卷的题目模板"——即 question_bank
2. **需要在 questions 中增加追踪字段**，记录题目是从题库哪个版本导入的
3. **需要版本链机制**，让同一题的多个版本可以共存
4. **需要共享机制**，让题目可以跨用户使用

### 3.3 设计决策

经过分析，确定了以下设计决策：

1. **导入创建独立副本**：从题库导入到问卷时，创建一份独立的题目副本。这样修改题库不影响已导入问卷（解决 P2）。
2. **版本链模式**：每道题库题目可以有多个版本，用 chain_id 串联，version 递增。每次修改创建新版本而非覆盖（解决 P3、P4）。
3. **定向分享机制**：支持"共享给指定用户"方式，共享的题目在「共享题目」tab 中展示（解决 P5）。
4. **bank_chain_id 追踪**：在 questions 中记录导入来源的 chain_id，用于查询使用情况和跨问卷统计（解决 P6、P7）。

## 5. MongoDB 设计（变化部分）

### 5.1 新增集合：question_bank

用于存储独立的题目模板，支持版本链管理：

| 字段         | 类型          | 说明                                  |
| ------------ | ------------- | ------------------------------------- |
| \_id         | ObjectId      | 题目主键                              |
| owner_id     | ObjectId      | 所属用户                              |
| type         | string        | single_choice/multi_choice/fill_blank |
| title        | string        | 题干                                  |
| options      | array         | 选择题选项                            |
| validation   | object        | 校验规则                              |
| chain_id     | ObjectId      | 版本链标识，首版本 chain_id = \_id    |
| parent_id    | ObjectId/null | 直接前驱版本                          |
| version      | int           | 版本号，从 1 递增                     |
| version_note | string        | 版本说明                              |
| shared_with  | array         | 共享目标用户 ID 列表                  |
| is_latest    | bool          | 是否为最新版本                        |
| created_at   | datetime      | 创建时间                              |
| updated_at   | datetime      | 更新时间                              |

### 5.2 questions 集合扩展字段

在第一阶段字段基础上新增 3 个可选字段，用于追踪题库导入关系：

| 字段          | 类型          | 说明                  |
| ------------- | ------------- | --------------------- |
| bank_item_id  | ObjectId/null | 导入来源的题库条目 ID |
| bank_chain_id | ObjectId/null | 导入来源的版本链标识  |
| bank_version  | int/null      | 导入时的题库版本号    |

### 5.3 新增索引

| 索引名                         | 定义                              | 用途                     |
| ------------------------------ | --------------------------------- | ------------------------ |
| idx_qbank_owner_latest_created | {owner_id, is_latest, created_at} | 查询我的题库（最新版本） |
| idx_qbank_chain_version        | {chain_id, version}               | 查询版本链历史           |
| idx_qbank_shared_latest        | {shared_with, is_latest}          | 查询共享给我的题目       |
| idx_question_bank_chain        | {bank_chain_id}                   | 跨问卷统计时查找关联题目 |

### 5.4 版本链设计

```
v1: chain_id=v1._id, parent_id=null,  version=1, is_latest=true
v2: chain_id=v1._id, parent_id=v1._id, version=2, is_latest=true  (v1.is_latest→false)
切换v1: v1.is_latest→true, v2.is_latest→false（直接切换，不创建新版本）
```

核心保证：每个版本是独立文档，修改不覆盖，历史不丢失。

## 12. AI 使用说明

### 12.1 AI 帮了什么

1. AI **修改了 question_bank 表结构和索引**
2. AI **生成了题库基础功能（保存/导入/分享）、题目版本功能（编辑/历史/恢复/版本隔离）、题目统计功能（使用情况/跨问卷统计）的全部前后端代码**
3. AI **更新了架构文档、数据库设计文档和 API 文档**
4. AI **编写了第二阶段全部测试用例**
5. AI **生成了初版第二阶段项目完成报告**

### 12.2 AI 做错了什么

1. **跨问卷统计将不同版本的结果错误合并**：AI 生成的跨问卷统计代码默认聚合全部版本数据，将不同版本的题目回答混在一起呈现，不符合按版本独立统计的需求，经指出后修正为必须指定版本才能统计
2. **版本恢复逻辑设计为创建新版本**：AI 生成的"恢复旧版本"功能会创建一份与旧版本内容相同的新版本，而非直接在已有版本间切换。我认为这不合理——用户期望的是回到旧版本本身，而不是创建副本。改为直接切换 is_latest 标记
3. **生成了不必要的题目公开功能**：AI 同时实现了题目公开和定向分享两种共享方式，我认为公开功能与定向分享有重复且不适合实际场景，将其删除
4. **跨问卷统计可视化布局有显示问题**：生成的前端跨问卷统计面板中选项分布进度条等元素布局不合理，需要单独修复
5. **编辑题库仅能修改标题和版本说明**：AI 生成的编辑表单只有标题和版本说明两个字段，无法编辑题型、选项、校验规则等核心内容，经指出后扩展为完整的题目编辑表单
6. **收藏题目未追踪来源问卷**：AI 生成的"收藏到题库"功能没有将原始题目标记为题库使用来源，导致使用情况列表中缺少收藏来源的问卷，经指出后新增 source_question_id 追踪机制
7. **自动化测试脚本使用了类型不友好的写法**：`(body or {}).get(...)` 三元表达式导致 IDE 大量类型警告

### 12.3 自己改了什么

1. **添加了 version_note 字段**：AI 设计的表结构中没有版本说明字段，我手动添加了 version_note，用于描述每个版本的变更内容
2. **删除了题目公开功能**：AI 同时生成了公开和定向分享两种共享方式，我删除了公开功能，只保留定向分享
3. **调整了版本恢复逻辑**：将"恢复旧版本 = 创建新副本"改为"直接在已有版本间切换 is_latest"，更符合用户直觉
4. **修复了跨问卷统计版本合并问题**：要求跨问卷统计必须指定版本，移除了"全部版本"统计范围
5. **添加了共享题目的收藏与移除功能**：共享题目新增"收藏到我的题库"和"移除共享"按钮
6. **修改了文档语言表述和格式**：AI 生成的文档内容在语言和格式上做了调整
7. **大幅度修改了项目完成报告**：AI 生成了初版报告，我对内容做了大幅修改

### 12.4 自己设计了什么

1. **题库功能需求分析与拆分**：将 8 条需求拆分为三批开发——基础功能（保存/导入/分享）、版本功能（编辑/历史/恢复/隔离）、统计功能（使用情况/跨问卷统计）
2. **question_bank 表核心字段设计**：确定了 owner_id、type、title、options、validation、version 等核心字段，以及 version_note 字段
3. **版本链的索引设计**：确定需要 idx_qbank_chain_version 索引来加速版本历史查询
4. **题目版本切换机制**：题库题目版本切换时，将 is_latest 置为 False，并切换到新的题目版本，将 is_latest 置为 True
5. **定向分享机制**：只支持共享给指定用户的方式，不支持公开
6. **跨问卷统计的聚合方案**：通过 bank_chain_id 关联所有使用同一题链的问卷题目，再按指定版本聚合答案
7. **共享题目的收藏与移除**：设计了共享题目的管理交互——支持收藏到自己题库（复制内容）和移除共享关系（从 shared_with 中移除）

## 14. 遇到的问题

1. **版本链设计的复杂度**：需要在不破坏旧数据的前提下引入版本管理，初期在"覆盖修改"与"新增版本"两种方案间犹豫
2. **导入后的追踪问题**：题目导入到问卷后需要知道来源，但第一阶段的 questions 文档没有来源追踪字段
3. **共享权限的边界**：用户共享题目后，共享题目是否应该包含所有版本？新创建的版本是否自动对已共享用户可见？
4. **跨问卷统计的性能**：需要跨多个问卷聚合答案，查询链路涉及 question_bank → questions → answers 三个集合
5. **自动化测试脚本的类型警告**：新增的测试方法使用了 `(body or {}).get(...)` 的三元表达式写法，IDE 类型检查器报错
6. **删除版本时 is_latest 的维护**：删除最新版本时需要将前一版本恢复为最新，如果只剩一个版本则整条链只剩空
7. **收藏来源追踪缺失**：收藏题目到题库后，原始问卷中的题目没有标记题库来源信息，导致使用情况列表中缺少来源问卷
8. **跨问卷统计范围过大**：跨问卷统计应该聚合全部版本数据，还是查看特定版本的统计？
9. **共享题目管理不足**：共享题目无法收藏到自己的题库，也无法从共享列表中移除不需要的题目

## 15. 如何解决

1. **选择"新增版本"方案**：每次修改题库题目创建新文档而非覆盖，保证历史不丢失。用 chain_id 串联同族题目，is_latest 标记最新版。对第一阶段数据无影响。
2. **在 questions 中新增 3 个可选字段**：bank_item_id、bank_chain_id、bank_version，全部通过 payload.get() 读取，默认 None，不破坏现有数据。
3. **共享作用于整条版本链**：共享时更新链上所有版本的 shared_with，新版本自动继承共享设置。这样用户共享后无需再次操作。
4. **通过 bank_chain_id 建立查询链路**：先通过 chain_id 找到所有关联的 questions，再按 survey_id 查找 answers 并聚合。新增 idx_question_bank_chain 索引加速查询。
5. **统一改为 if isinstance 分支模式**：将 `(body or {}).get(...)` 改为 `data = {}; if isinstance(body, dict): data = body.get("data") or {}` 的分支写法，消除类型警告。
6. **删除时检查并恢复前一版本**：在 delete_question_bank_item 中，如果删除的是 is_latest 版本，自动查找前一版本并设为 is_latest=true。
7. **收藏来源追踪**：QuestionBankCreateSerializer 新增 source_question_id 字段，create_question_bank_item 在创建题库条目后自动更新原始题目的 bank_item_id、bank_chain_id、bank_version 字段，使原始问卷出现在使用情况中。
8. **跨问卷统计必须指定版本**：移除"全部版本"统计范围，get_bank_cross_stats 在未传入 version_item_id 时返回错误，前端默认选中当前版本并隐藏"全部版本"按钮。
9. **共享题目收藏与移除**：共享题目新增「收藏」按钮（调用创建题库接口复制内容到我的题库）和「删除」按钮（后端新增 remove-shared 端点，将用户从 shared_with 数组中移除）。
